## Overview
The project employs a minimalist, zero-dependency build and testing strategy centered around the native Node.js runtime. It avoids complex build tooling (such as Webpack, Vite, or Make) in favor of direct execution and native browser capabilities for its Progressive Web App (PWA) architecture.

## Build System Components

### 1. Package Management & Scripts
- **Tool:** `npm` (Node Package Manager)
- **Configuration:** `package.json`
- **Strategy:** The project is configured as `private: true` with no external dependencies listed. This indicates a "vanilla" JavaScript approach where all logic is contained within local files (`index.html`, `sw.js`, etc.).
- **Scripts:**
  - `npm test`: Executes the test suite using Node's built-in test runner (`node --test`).

### 2. Continuous Integration (CI)
- **Platform:** GitHub Actions
- **Configuration:** `.github/workflows/test.yml`
- **Pipeline Flow:**
  1. **Trigger:** Runs on every push to any branch and on all pull requests.
  2. **Environment:** Uses `ubuntu-latest` with Node.js version 20.
  3. **Steps:**
     - Checks out the repository code.
     - Sets up the Node.js environment.
     - Executes `npm test`.
- **Dependency Strategy:** The CI pipeline explicitly skips dependency installation (`npm install`) because the project has no `node_modules` dependencies. Tests rely solely on the Node.js standard library.

### 3. Testing Framework
- **Runner:** Node.js Built-in Test Runner (`node:test`)
- **Test Files:** Located in the `test/` directory (e.g., `test/logic.test.js`).
- **Execution:** Tests are run directly via the CLI without additional test harnesses like Jest or Mocha.

## Deployment & Artifact Management
- **Static Hosting:** As a PWA, the application is designed to be served as static files (`index.html`, `manifest.json`, `sw.js`). There are no compiled artifacts (bundles, minified JS) generated by a build step.
- **Service Worker:** `sw.js` handles offline caching and asset management client-side, replacing the need for server-side build caching strategies.
- **No Containerization:** There are no `Dockerfile`s or container orchestration configurations, consistent with a static frontend deployment model.

## Developer Conventions
1. **Zero-Build Development:** Developers can run and test the application by serving the root directory directly. No compilation step is required.
2. **Native Tooling Only:** Use only native Node.js APIs for testing and logic. Avoid introducing external libraries unless absolutely necessary, as this would break the current zero-dependency CI flow.
3. **CI Stability:** The CI pipeline is lightweight and fast. Any new scripts added to `package.json` must be compatible with the `ubuntu-latest` runner and Node 20 environment.
4. **Versioning:** The project is currently at version `1.0.0`. Version bumps should be managed manually in `package.json` as there is no automated release pipeline configured.
